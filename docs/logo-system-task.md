# Logo System Task — Clean, Reliable Tool Imagery

## Why this task exists

Current tool imagery is visually inconsistent:
- some assets are true vendor/product logos from primary sources
- some are placeholder/generated letter-badge style SVGs
- some product lines likely use mixed visual conventions across vendors

Even though the data model is consistent (`logoUrl` is populated for all current tool records), the *asset quality and provenance* are not consistent enough for a clean production presentation.

## Current audit snapshot

Checked on 2026-04-18.

- Tool records in `data/tools.json`: **41**
- Tool records missing `logoUrl`: **0**
- Logo files under `public/logos/`: **46** (including helper files like `.gitkeep` / `README.md`)
- Existing provenance note: `public/logos/README.md`
- Current rendering component: `components/logo-badge.tsx`
- Current behavior: if `logoUrl` exists, render the image; if not, fall back to first-letter badge

## Observed root problem

The issue is **not** primarily missing wiring in code.
The issue is that the asset library itself mixes:
- official product/vendor marks
- architecture/service icons
- placeholder/generated badges
- uneven visual styles across vendors/categories

That creates a low-trust, messy feel even when technically every record has a logo path.

## Goal

Create a **clean, reliable, maintainable logo system** with explicit rules.

## Requirements for the solution

### 1) Provenance rules
Define allowed asset sources, in priority order:
1. official vendor/product brand assets or icon libraries
2. official OSS project/repo branding assets
3. official docs/media kits with reusable marks
4. controlled fallback asset generated by project rules

Disallow ad-hoc third-party scraped logos unless there is a documented reason and explicit note.

### 2) Fallback rules
If no defensible official product mark exists, do **not** silently mix arbitrary pseudo-logos.
Use a deliberate fallback strategy, e.g. one of:
- standardized vendor family glyph + product label
- standardized neutral monogram system
- text-first pill / badge treatment

This should be consistent, not improvised per file.

### 3) Asset standards
Define:
- preferred format: SVG when available from official source
- target framing / padding rules
- light/dark background expectations
- minimum viewport legibility
- handling for monochrome vs multicolor marks
- whether architecture icons are acceptable substitutes for product marks

### 4) Data/model rules
Decide whether current `logoUrl` is enough or whether the schema should be extended with metadata such as:
- `logoKind`: `official-product` | `official-vendor` | `service-icon` | `project-logo` | `fallback`
- `logoSourceUrl`
- `logoNotes`
- `logoReviewedAt`

A metadata layer would make the system auditable and maintainable.

### 5) UI rules
Review whether all logos should be rendered in the same box treatment.
Potential improvements:
- normalized background treatment
- consistent padding per asset kind
- optional badge/ring for fallback assets
- distinct styling for fallback vs official mark so it reads intentional, not broken

### 6) Maintenance workflow
Need a repeatable process for adding/updating logos:
- where to source from
- what evidence to record
- how to note placeholders/fallbacks
- how to review replacements later

## Recommended execution plan

### Phase 1 — inventory + classification
For every platform/tool logo, classify as:
- official product logo
- official vendor/service icon
- OSS project logo
- placeholder/fallback
- questionable/needs review

### Phase 2 — define system
Write a short design spec covering:
- accepted source hierarchy
- approved fallback style
- schema additions if needed
- rendering standards

### Phase 3 — implement
- add metadata if justified
- normalize rendering rules in `components/logo-badge.tsx`
- replace weakest assets first
- document provenance and fallback status clearly

### Phase 4 — validate
- visual review across all category pages
- dark/light contrast review
- mobile legibility review
- ensure no broken logos/fallback regressions

## Initial recommendation

Do **not** jump straight into mass asset swapping.
First produce a small classification table and decide one fallback standard.
That is the reliable path.
